# 要求 1：AI Agent 钱包的用户画像与使用场景

## 1. 用户画像

### 1.1 AI 原生开发者

这类用户平时已经在使用 Codex、Claude Code 等 AI Agent，希望直接通过自然语言完成钱包操作，而不是手动点钱包插件或输入长地址。

他们最关心的是：

- 能否直接说“向某某转 0.01 ETH”
- 能否在执行前看到金额、地址、手续费
- 能否方便地查看提案状态和交易历史

### 1.2 轻度 Web3 用户

这类用户不熟悉 Gas、RPC、链 ID、私钥等底层概念，但能理解“转给谁”“转多少”“确认或取消”这种简单流程。

他们最关心的是：

- 操作方式是否接近日常对话
- 确认信息是否清楚易懂
- 不需要直接接触复杂链上参数

### 1.3 本地操作员（用户本人 或 用户授权的帐户管理员

这类用户负责配置和维护钱包系统的运行环境，而不是日常发起转账。他们拥有更高权限，用于控制系统的安全边界。

他们最关心的是：

- 私钥、白名单、策略等敏感配置必须与 AI Agent 隔离
- Agent 只能在受控范围内执行操作
- 可以方便地调整测试环境（如地址簿、限额、开关等）
- 所有高风险操作（如执行转账）必须可控、可审计

在当前项目中，该角色通常通过本地控制台或 Web 管理界面完成配置和审批操作，从而实现人机分离（Human-in-the-loop）的安全机制。

## 2. 使用场景

### 2.1 自然语言发起转账

用户直接对 Agent 说：

- 帮我向 0x1234567890AbcdEF1234567890aBcdef12345678 这个地址转 0.01 ETH

这是当前项目最核心的场景。Agent 负责创建 proposal，展示金额、地址和手续费，用户确认后再执行。

### 2.2 向固定联系人转账

用户不希望每次都输入完整地址，而是希望直接说联系人名称或别名。例如：

- 向小明转 0.01 ETH
- 帮我给 Zoey 转 0.02 ETH

这个场景说明地址簿和联系人解析对 AI Agent 钱包很重要，因为它能降低输错地址的风险。

### 2.3 执行前确认或取消

用户不希望一句话就直接完成转账，而是希望在执行前再核对一次关键信息。

因此，实际流程应该是：

1. Agent 创建 proposal
2. Agent 展示金额、地址、手续费
3. 用户回复“确认”或“取消”
4. 系统再决定执行或终止（只有用户确认才能执行转账）

这个场景体现了 AI Agent 钱包和普通命令执行器的区别，即资金操作必须有明确确认过程。

### 2.4 查询账户信息

用户在真正发起转账前，通常会先确认当前钱包的基本信息，例如：

- 当前余额是多少
- 当前钱包地址是什么
- 当前使用的是哪条链

因此，查询账户信息也是一个核心场景。用户可以先通过 Agent 了解当前钱包状态，再决定是否继续发起转账。

### 2.5 查询提案和交易历史

用户除了发起转账，还需要知道：

- 这笔 proposal 现在是什么状态
- 最近一笔交易是否执行成功
- 查看最近的10个交易

因此，查询 proposal、交易状态和历史记录也是核心场景之一。

### 2.6 展示收款信息

用户不只需要“转出”，也需要“收款”。因此，Agent 还应能帮助用户生成适合展示或转发的收款信息。例如：

- 把我的收款信息发给我

### 2.7 人工后台维护测试环境

本项目是一个本地 Demo，因此还存在一个重要场景：由用户本人或其授权的人类管理员维护测试环境。包括：

- 调整模拟余额
- 修改白名单
- 维护地址簿
- 修改策略和权限开关

它决定了 Agent 钱包能否在受控边界内稳定运行。

## 3. 小结

当前项目最核心的用户可以概括为三类：

- 使用 Agent 发起钱包操作的开发者
- 希望用自然语言完成简单转账的轻度 Web3 用户
- 负责本地配置和安全边界的用户本人或其授权的人类管理员

最核心的使用场景则包括：

- 自然语言转账
- 联系人化转账
- 执行前确认 / 取消
- 查询账户信息
- 查询提案和交易历史
- 展示收款信息
- 人工后台维护环境

因此，AI Agent 钱包的重点不是单纯“能转账”，而是让 Agent 在受控流程里协助用户完成钱包操作。
