# 要求 2：AI Agent 钱包需要重点解决的问题

## 1. 问题一：权限边界如何划分

AI Agent 钱包和普通钱包最大的不同，不是“能不能转账”，而是“哪些能力可以交给 Agent，哪些能力不能交给 Agent”。

如果权限边界不清楚，就会出现两个高风险问题：

- Agent 可能直接接触私钥、白名单、策略开关等敏感配置
- Agent 可能绕过人工控制，执行本不该执行的操作

因此，AI Agent 钱包首先要解决的问题就是权限分层。

在当前项目中，这个问题的设计思路是：

- Agent 只负责查询、创建 proposal、确认 proposal、执行受控转账
- 私钥、白名单、地址簿、模拟余额、策略开关等敏感配置只允许人工后台处理
- 通过 MCP 工具层和 Operator Console 把“使用钱包”和“管理钱包”分开

## 2. 问题二：如何避免一句话直接触发资金操作

AI Agent 的优势是自然语言交互，但这也是钱包场景里最大的风险来源之一。

如果用户只说一句“帮我转账”，系统就直接执行，那么很容易出现：

- 地址输错
- 金额理解错误
- 用户还没来得及核对就已经发生资金动作

因此，AI Agent 钱包必须把“发起请求”和“真正执行”拆开。

在当前项目中，这个问题的设计思路是：

- 所有转账先创建 proposal
- 执行前必须展示金额、地址、手续费
- 用户明确回复“确认”后，系统才允许进入执行阶段
- 如有需要，还可以进一步开启本地二次授权模式

也就是说，系统不能把自然语言直接等同于最终资金指令，而必须经过确认流程。

## 3. 问题三：如何确保交易意图在执行过程中不被篡改（Intent Integrity）

AI Agent 钱包的一个关键风险是：用户确认的是一笔交易，但系统真正执行时，核心参数可能已经不是原来的那一笔。

例如：

- proposal 阶段是转给地址 A
- 执行阶段却变成转给地址 B

这类问题比一般的自然语言解析错误更严重，因为它直接影响真实的资金去向。

因此，AI Agent 钱包必须保证：用户确认的核心交易意图，在执行阶段仍然保持一致。

在当前项目中，这个问题的设计思路是：

- 在 proposal 创建时生成 `intent_hash`
- `intent_hash` 只锁定稳定业务字段，例如收款地址、金额、链 ID
- 在执行前重新计算 `intent_hash`
- 如果执行前计算结果与 proposal 记录不一致，则直接拒绝执行
- gas、nonce 这类动态执行参数不纳入业务意图哈希

这样做的意义是：

- 锁定真正需要用户确认的业务意图
- 防止 proposal 和 execute 之间被偷偷改参
- 同时保留执行阶段必要的动态参数调整空间

## 4. 小结

设计 AI Agent 钱包时，最需要优先解决的 3 个问题是：

1. 权限边界如何划分
2. 如何避免一句话直接触发资金操作
3. 如何确保交易意图在执行过程中不被篡改（Intent Integrity）

这三个问题分别对应当前项目里的三条核心设计主线：

- 权限隔离
- proposal 确认流程
- intent integrity 校验

它们共同决定了这个项目为什么不是一个普通钱包 Demo，而是一个面向 AI Agent 场景的钱包 Demo。
